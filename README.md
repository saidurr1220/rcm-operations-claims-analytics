# RCM Operations Analytics Dashboard

A full-stack web application for healthcare Revenue Cycle Management (RCM) analytics, built as a portfolio project demonstrating real-world data analysis and operational workflows.

## Overview

This application provides comprehensive analytics for RCM operations, including:

- Claims KPI tracking (denial rates, approval rates, processing times)
- Revenue analysis by payer and encounter type
- Patient financial summaries
- Data quality monitoring and issue detection

**Note:** This project uses synthetic healthcare data (Synthea + Kaggle datasets). No real patient data or PHI is included.

## Tech Stack

### Backend

- **Node.js** (v18+) with Express
- **PostgreSQL** (v14+) for data storage
- **Raw SQL** queries for analytics (demonstrates SQL proficiency)
- **csv-parse** for data import

### Frontend

- **Next.js 14** (App Router) with TypeScript
- **React 18** with functional components and hooks
- **CSS Modules** for styling

## Data Sources

1. **Synthea Clinical Data** (patients, encounters, conditions, immunizations)

   - Synthetic patient records generated by Synthea
   - Download from: https://synthea.mitre.org/downloads

2. **Kaggle Health Insurance Claims Dataset**
   - Enhanced claims data with denial reasons, processing times
   - Download from: https://www.kaggle.com/datasets (search for health insurance claims)

Place all data files in the `/data` directory before running import scripts.

## Setup Instructions

### Prerequisites

- Node.js v18 or higher
- PostgreSQL v14 or higher
- npm or yarn

### 1. Clone the Repository

```bash
git clone <your-repo-url>
cd rcm-operations-analytics-dashboard
```

### 2. Set Up PostgreSQL Database

```bash
# Create database
createdb rcm_analytics

# Or using psql
psql -U postgres
CREATE DATABASE rcm_analytics;
\q
```

### 3. Configure Backend

```bash
cd backend
npm install

# Copy and edit environment variables
cp .env.example .env
# Edit .env with your PostgreSQL credentials
```

Example `.env`:

```
DATABASE_URL=postgresql://username:password@localhost:5432/rcm_analytics
DB_HOST=localhost
DB_PORT=5432
DB_NAME=rcm_analytics
DB_USER=your_username
DB_PASSWORD=your_password
PORT=3001
NODE_ENV=development
DATA_DIR=../data
```

### 4. Initialize Database and Import Data

```bash
# Create tables and indexes
npm run init-db

# Import CSV/TXT files
npm run import-data

# Or run both commands
npm run setup
```

### 5. Start Backend Server

```bash
npm start
# Server runs on http://localhost:3001
```

### 6. Configure Frontend

```bash
cd ../frontend
npm install

# Copy environment file
cp .env.local.example .env.local
# Edit if your backend runs on a different port
```

### 7. Start Frontend Development Server

```bash
npm run dev
# Frontend runs on http://localhost:3000
```

### 8. Access the Application

Open your browser to `http://localhost:3000`

## API Endpoints

### KPIs

- `GET /api/kpis/overview` - Overall KPIs (claims, denial rate, approval rate)
- `GET /api/kpis/denials-by-payer` - Denial statistics by payer
- `GET /api/kpis/denials-by-claim-type` - Denial statistics by claim type
- `GET /api/kpis/monthly-trends` - Monthly claim trends

### Revenue

- `GET /api/revenue/by-encounterclass` - Revenue by encounter class
- `GET /api/revenue/by-payer` - Revenue by payer
- `GET /api/revenue/summary` - Overall revenue summary

### Patients

- `GET /api/patients/summary` - Patient list with financial data
- `GET /api/patients/:id` - Individual patient details
- `GET /api/patients/:id/encounters` - Patient encounters
- `GET /api/patients/:id/claims` - Patient claims

### Quality Assurance

- `GET /api/qa/issues` - Data quality issues
- `GET /api/qa/summary` - QA issue summary

## Dashboard Pages

1. **Overview** (`/`) - High-level KPIs and monthly trends
2. **Denials & Revenue** (`/denials`) - Denial analysis by payer/type, revenue breakdown
3. **Patients** (`/patients`) - Patient demographics and financial summaries
4. **QA Issues** (`/qa`) - Data quality problems and validation checks

## Key Features

### RCM Analytics

- Denial rate tracking by payer and claim type
- Approval rate monitoring
- Average processing time analysis
- Revenue analysis by encounter class
- Patient financial summaries

### Data Quality Checks

- Negative or zero claim amounts
- Cost mismatches (claim cost < base cost)
- Coverage exceeding claim amounts
- Missing critical fields
- Implausible patient ages
- Missing payer assignments
- Long processing times (>90 days)
